# W7（2026-06-23 – 06-29）

## 本週主軸

- 把 CloudDrive 開發環境完整建起來，熟悉前端架構。
- 為 AI 助理推進四項功能：模型選擇、自建 skill 進對話、結構化輸出修復、多組具名連線。
- 建立執行稽核（execution audit）方法：查核 AI 宣稱的操作是否真的執行。

## 完成項目

### 1. 開發環境建置與前端架構導讀

把 CloudDrive 在本機完整跑起來（Postgres 容器、後端、前端 dev server），並通讀前端架構。

### 2. 助理功能：模型選擇

對話框加模型下拉，選定即只用該模型、不在失敗時自動改用其他模型（fallback）；負責呼叫本機模型的程式（`OllamaLLMClient`）加上 5 秒連線逾時，連不到的本機模型立即失敗（原本每次對話都要等滿 300 秒逾時，是做這個功能的直接動機）；對話 API 的錯誤分類（接不到／憑證被拒或額度用盡／其他）。

### 3. 助理功能：自建 skill 可在對話中使用（chat-skills）

使用者自建的 skill（如 `compress_to_zip`）可被規劃器（planner——把使用者訊息轉成執行計畫的元件）選用——一律列為會改動資料的 write 級（必經使用者確認才執行）、在沙盒中執行、以勾選檔案帶入操作目標（多選時逐檔展開）、每個 skill 預設關閉需手動啟用。

**問題 1：對話框輸入「用我的 skill 壓縮檔案」，助理回「技能集中沒有這個功能」**
- 原因：規劃器的技能清單只載入 13 個內建 skill；自建 skill 存在資料庫裡（一段程式文字），原本只能經右鍵選單觸發沙盒執行，從未被放進規劃器可選的技能清單。
- 解決：把已安裝且允許進對話的自建 skill 註冊進技能清單（registry），執行時橋接到沙盒；名稱與內建衝突則跳過並提示改名。「對哪個檔案操作」不靠 LLM 猜，由前端勾選檔案帶入、逐檔展開步驟。

### 4. 助理功能：外部模型結構化輸出修復（json_schema）

外部模型規劃時改用 json_schema——把 JSON 結構規格隨請求送給模型、強制輸出符合規格，而不是只在 prompt 裡懇求。

**問題 1：選 Gemini 後助理只會閒聊——回「好的，我會將選取的檔案壓縮成 zip」之類的話，但確認畫面不出現、什麼都沒執行**
- 原因：規劃器要求 LLM 回固定 JSON `{"reply", "steps":[...]}`。本機 Gemma 靠 prompt 就照做，Gemini 不照——三種模式實測：
    1. 純文字模式：回自然語言，解析失敗變閒聊。
    2. `json_object` 模式：回 JSON 但欄位名錯。
    3. `json_schema` 模式：完全正確。

    另有前置缺陷：規劃器組 prompt 時沒告知「使用者已勾選 N 個檔案」，Gemini 一直反問要對哪個檔案操作。
- 解決：對話介面加格式規格參數（`response_format`），由規劃器 → 模型路由（ModelRouter，決定呼叫本機或外部模型的元件）→ 外部模型呼叫程式一路轉交；規格不加 strict 模式（與開放式參數欄位衝突，服務端會拒收）；規劃器加系統訊息告知選取數量。範圍限定「只動規劃器、只動外部路徑」——這個限定在 W8 被證明留下缺口（本機路徑漏傳，見 W8）。

### 5. 助理功能：多組具名模型連線

以新資料表取代單一憑證：每筆連線記名稱（label）、服務類型（kind）、位址（base_url）、模型名（model）、金鑰（key），助理下拉列出本機＋每筆連線，額度用完可直接切換另一筆。

### 6. 外部模型實測（Gemini、Ollama cloud）

實測兩家服務的可用組態與限流行為，把可用組態與模型目錄記進筆記。

**問題 1：測試中 Gemini 開始回 429 RESOURCE_EXHAUSTED；Ollama cloud 連測多次後所有模型回 401 Unauthorized**
- 原因：Gemini 免費方案每天每模型只有 20 次請求（各模型額度獨立），當日測試加診斷就用光；Ollama cloud 免費 key 有限流，401 是限流不是設定錯。另有組態陷阱：Ollama cloud 要用 `openai_compatible` kind＋`https://ollama.com/v1`（原生 `/api/chat` 路徑會 404 且不保證 json_schema）。
- 解決：換模型取得獨立額度（`gemini-2.5-flash-lite`）、等額度重置。長期結論：穩定測試需要付費、或多備來源輪替。

### 7. 執行稽核：AI 回報與實際執行的查核方法

助理宣稱「刪好了」但檔案原封不動——本項建立查核方法，確認宣稱的刪除從未發生。

**問題 1：請助理「把選取的檔案丟到垃圾桶」，回覆「做好了」，實際什麼都沒做**
- 原因：規劃器回的 `reply` 是給人看的話、`steps` 才是要執行的動作，兩者是同一次 LLM 輸出的不同欄位——LLM 可能 reply 寫「刪好了」但 steps 是空的，此時程式直接回覆那句話、不執行也不留紀錄。查核確認丟垃圾桶的工具本身存在，是規劃器沒把「刪除」對應到它——LLM 規劃失敗，而非缺工具。
- 解決：本週的產出是查核方法而非修復——用資料庫指令列工具（psql）對三張表交叉查核：計畫表 `assistant_workflows`（7 筆全是壓縮、無任何刪除計畫）、執行紀錄表 `assistant_workflow_runs`（只有真的執行過才會有紀錄）、操作日誌表 `activity_logs`（垃圾桶事件 0 筆）。確立法則：**LLM 的自述不能當證據，真做了必留資料庫稽核紀錄；兩者都沒有＝沒做**。（「回報必須以實際執行結果為依據」在 W8 以「回覆由程式從執行結果組合」的機制（honest reporting）根治。）

### 8. 檔案預覽／下載缺陷診斷（repo 既有 bug，本週僅診斷）

**問題 1：點檔案「預覽」顯示 Failed to load preview，點「下載」回 401 Not authenticated**
- 原因：兩個獨立缺陷——
    1. 前端請求的預覽資訊路徑在後端不存在，回 404。單元測試沒抓到，是因為前端測試用假後端工具（MSW）模擬了這條後端根本沒有的路徑——假後端把前後端規格不一致蓋住了。
    2. 下載與預覽用連結直接開啟檔案網址，這種請求瀏覽器不會附上登入憑證；而本專案的登入憑證（access token）只存在前端程式記憶體、不放 cookie，後端收不到憑證回 401。
- 解決：本週先不修——用 git 比對確認是專案本身的 bug（與遠端版本一字不差），並用附上憑證的手動請求得到 200，證明後端端點本身正常。修法方向記入待辦。（隊友於 6/29 實作修復：改由程式帶憑證抓取檔案內容再顯示（blob 載入），7/13 隨 PR #10 合併。）

### 9. Codex 訂閱制接入分析（外部模型的一條路徑）

助理的外部模型可走兩條路徑：路徑 A 用 Codex 訂閱額度（憑證是 `codex login` 產生的 token）、路徑 B 用 OpenAI API key（按量計費）。本項分析路徑 A 是否適合一般使用者。（對應決策 DEC-026；完整分析見 cloud_drive 的 [tasks/external-model.md](https://github.com/billwu101/CloudDrive/blob/main/doc/tasks/external-model.md)。）

**問題 1：路徑 A 要一般使用者自己裝 Node／CLI、跑 `codex login` 做 OAuth、再把產生的 token 手動貼進網站——這是工程師等級的流程，非技術使用者幾乎做不到。理想的「網頁內點一下就授權」（像 Sign in with Google）在雲端架構下做不到**
- 原因：能扣訂閱額度的只有 Codex 的 OAuth token，它的授權走本機流程——回呼固定發到 `localhost:1455`，必須和開瀏覽器的那台機器同機才接得到；雲端後端在另一台機器，接不住這個回呼。回呼位址又寫死、無法申請自己的憑證繞過，屬服務條款灰色地帶。「本機收到再交給雲端」技術可行，但那需要在使用者電腦裝一支常駐程式，等於回到裝 CLI 的門檻。
- **解決（結論）**：路徑 A 定位為技術型進階使用者的選配、不主推；一般大眾上線應走「由 App 統一提供模型＋用量控制」，或「使用者自帶 API key」（路徑 B）。「網頁內點按授權」只有在後端與瀏覽器同機的「本機自架版」才可能實作。

## 待釐清

- **連線網址未設防護（SSRF 風險）**：「多組具名連線」的 base_url 可以填任意網址，後端會直接對它發請求——惡意使用者可填內網位址，讓伺服器代為存取內部服務（SSRF，伺服器端請求偽造）。防護方向：網址白名單、或僅允許對外的 https 位址。
- **選檔與計畫畫面顯示內部 id 而非檔名**：系統內部以 UUID 識別檔案（檔名可重複、也可能被改名，id 唯一且穩定，所以操作一律用 id），但顯示層直接把 id 露給使用者是第一版未完成的部分——選取狀態只存了 id 沒存名稱，要顯示檔名需擴充狀態或查快取，待改善。
- **計畫確認畫面的前端流程尚未實際走過**：寫入／破壞性計畫會標記為待確認、由使用者在前端按下確認才執行；本週只從後端確認了這個機制存在，還沒實際操作過前端的確認畫面。
