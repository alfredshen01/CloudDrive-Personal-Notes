# W8（2026-06-30 – 07-06）

## 本週主軸

- 把結構化輸出的格式約束（constrained decoding，約束解碼）補到本機模型路徑，不再只有外部模型受保護。
- 失敗處理工程：回覆改由實際執行結果組成（honest reporting）＋單一步驟失敗不再中止整批計畫。

兩件理解性工作：走讀專案的 GitHub Actions 自動化流程（每次推送程式碼時自動執行測試、型別檢查、lint 的機制），理解各項檢查的接線與觸發條件；盤點後端與 AI 助理（harness）相關的既有程式碼，釐清分層——訊息入口 service → 規劃器 planner（把使用者訊息轉成執行計畫）→ 模型路由 ModelRouter（決定呼叫本機或外部模型）→ 步驟執行器 WorkflowExecutor（逐步執行計畫）。這份盤點直接支撐第 2 項（本機模型的格式約束補全）的修復定位。

### 2. 本機模型的格式約束補全（constrained decoding）

背景：AI 助理要求 LLM 回覆固定格式的 JSON 計畫。外部模型（如 Gemini）已在上週用 json_schema（把 JSON 結構規格隨請求送給模型、強制輸出符合）保護；本週把同樣的保護補到本機 Ollama 路徑。開發方式為 TDD（測試驅動開發）：先寫一個預期會失敗的測試，再修改程式讓它通過，確保每個修復都有測試佐證。

- **問題（一）**：負責呼叫本機模型的程式（`OllamaLLMClient`）收到完整的 json_schema 規格後，實際送給模型的請求只寫了「回 JSON」——具體的欄位、型別、必填規格整個被丟棄。格式跑掉時只能事後把錯誤丟回去請模型重寫（慢、耗 token）。
- **原因**：兩層——
    1. 舊程式把格式規格當成「開／關」的開關，規格內容沒看就丟了。
    2. 形狀不匹配：上游流通的是 OpenAI 的包裝格式 `{"type":"json_schema","json_schema":{...}}`，但 Ollama 只認未包裝的裸 JSON Schema，原封轉送本來就不可行，必須先拆掉包裝。
- **解決**：新增拆包裝的轉換函式（`_to_ollama_format()`）；三個「先失敗、修復後通過」的測試鎖住行為（含「沒有指定格式時維持原行為」——一般聊天不受影響）。兩道既有防線明確保留：格式規格只保證形狀、擋不住語意錯誤（例如編造不存在的功能），所以事後的計畫驗證與重試機制不拆；schema 維持手寫、不換成自動產生（自動產生的版本約束更弱），另寫一個比對測試確保手寫版不會與程式內的定義脫節。

- **問題（二）**：寫「規劃器每次呼叫模型都必須帶格式規格」的防護測試，測試沒有通過——檢查發現模型路由（ModelRouter）呼叫本機模型時，根本沒把格式規格轉交下去。
- **原因**：上週的修復範圍限定「只動外部路徑」，本機路徑從頭到尾收不到規格——就算問題（一）修好了拆包裝，也沒東西可拆。斷點在中間層；先前靠讀程式碼推測、判斷錯了位置，先寫一個會失敗的測試直接把斷點逼出來。
- **解決**：本機呼叫補上轉交，一行修掉。這是 TDD 價值的直接示範：推測會誤導，失敗的測試不會。

### 3. 回覆改由執行結果組成＋有限度重規劃（DEC-029）

（DEC-029 是專案決策紀錄的編號——重要架構決策都以 DEC-編號寫成正式紀錄，下同。）

分三級：

- **第 0 級**：執行失敗時的回覆由程式從每一步的執行結果紀錄（`StepResult`）組合，不再使用 LLM 在規劃時預先寫好的回覆文字。
- **第 1 級**：對話的快速路徑在執行失敗時，把真實執行結果餵回規劃器重規劃一次。
- **第 2 級**（讓 LLM 自主多輪循環的 agentic loop）：評估後決定不做，理由完整寫入決策紀錄。

- **問題**：計畫執行失敗，使用者收到的仍是成功語氣的回覆；確認執行與重跑路徑更是固定回覆 "Workflow executed."，即使內部狀態已是失敗。
- **原因**：LLM 產生計畫時會同時寫一句給使用者看的回覆（`plan.reply`）。那句話寫在任何步驟執行之前，本質是**預測**；程式卻在執行完之後把它原樣回給使用者——把預測當成報告，問題出在架構，不是模型說謊。
- **解決**：失敗時的訊息改由程式從執行結果組合——哪一步失敗、原因、之前完成了哪幾步、之後哪些沒有執行；三條執行路徑統一為「全部成功→原回覆，有失敗→一律用執行結果組合的報告」。重規劃加兩道護欄：只允許純讀取的計畫（不能升級成會改資料的操作，否則事前確認機制被繞過）、只重試一次絕不循環。重規劃只開放給對話快速路徑：「確認執行」的授權範圍是使用者看過的那份具體清單、「重跑」的授權範圍是固定配方，兩者都不允許偷換步驟。

### 4. 步驟失敗隔離（DEC-030）

一個步驟失敗，不再讓整批計畫全部中止。每一步的結果分三種狀態：成功（ok）、失敗（failed）、被擋（skipped——依賴的前一步沒成功）；每一步都必須留下一筆結果紀錄。前端以不同圖示區別顯示。同時正式決定：步驟維持一個接一個執行，「多步驟同時並行」延後，理由記入決策紀錄。

- **問題**：壓縮 5 個檔案，第 1 個失敗，其餘 4 個互不相關的步驟完全沒做，也沒有留下任何紀錄。
- **原因**：執行器遇到第一個錯誤就整批中止；「有依賴關係才該被擋」和「無關步驟」沒有被區分。
- **解決**：合併兩種依賴判斷——步驟明寫的依賴欄位（`depends_on`）與「引用前一步輸出」的參數（`from_step`）；被擋的步驟記為 skipped 並註明依賴哪一步，失敗的步驟記錄後繼續執行其餘步驟。並行延後的理由：所有功能共用同一個資料庫連線、技術上不可同時執行，這是最硬的前置；且隔離解決的是正確性、並行只縮短等待時間，價值不對等。

### 5. 防止重複生成修正兩輪（DEC-031／DEC-032）

「重複生成迴圈」指本機模型生成時掉入同樣內容不斷重複的循環，直到逾時。

- **問題（一）**：某些請求（如「查剩餘容量」）必定讓本機模型 gemma4 掉入重複生成迴圈，每次都等滿 300 秒逾時才失敗。
- **原因**：結構化請求把 temperature（生成隨機度）設為 0，模型每一步都選機率最高的字（greedy decoding），在特定輸入下會掉入完全確定性的重複循環——同樣的輸入 100% 重現。
- **解決**：DEC-031 做兩件事——
    1. 設生成長度上限：就算掉入重複生成迴圈也在限制內失敗，不再等滿逾時。
    2. 結構化請求改用低而非零的 temperature：微量隨機性打破循環，格式仍由規格約束保證。

    原本因重複生成迴圈逾時而失敗的真模型整合測試（實際呼叫本機模型的端到端測試）重複執行 40 次全數通過。

- **問題（二）**：量測發現部分失敗案例，是計畫裡寫了不存在的功能名稱，靠事後的計畫驗證才攔下、觸發重問。
- **原因**：格式規格只約束「功能名稱是一個字串」，沒約束「必須是哪些字串」——編造的名稱在格式上是合法的。
- **解決**：DEC-032——把功能名稱做成封閉選項清單（enum，依當下可用功能動態組進規格），編造的名稱從生成層就不可能出現。附帶發現：計畫驗證漏了檢查「依賴了不存在的步驟」，一併補上。

### 6. temperature 掃描工具 temp_sweep＋首次真模型量測

建立 temperature 掃描工具（在 eval 任務清單中編號 E7）：對代表性請求，在多種 temperature 設定下各重複執行多次，統計通過率、重複生成迴圈率與延遲——用實際量測取代「temperature 該設多少」的推測。首次量測選了 4 個代表案例、4 種 temperature、每個組合各跑 5 次，結果首次以數據確認：**破壞性規劃（刪除、覆蓋類請求）是模型最不可靠的類別**——依 temperature 不同，通過率只有 0% 到 40%，且重複生成迴圈與編造功能名稱兩種失敗並存。這個數據直接排定了後續修正的優先順序。

- **問題**：首輪數據出現大量 401 未授權錯誤；另有一次後端中途退出沒被偵測，整批數據作廢。
- **原因**：401 是登入憑證過期造成的假失敗（長時間掃描超過 access token 的有效期），不是模型問題；後端子行程死掉時，工具的就緒檢查沒有偵測行程是否存活。
- **解決**：每個案例執行前重新登入；就緒檢查加上行程存活偵測。教訓：**量測工具本身也要驗證**，否則資料驅動的改進會建立在髒數據上。
