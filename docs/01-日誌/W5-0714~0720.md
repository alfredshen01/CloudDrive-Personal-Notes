## 遇到的問題

**問題 1：第一輪審查的「通過」判定，證據撐不起結論**
- 緣由：V0.2 逐條審查第一輪完成後，後設稽核發現多個「通過」判定的證據只到「檔案存在／grep 命中」層級；補查四次、兩次改寫結論——漏看 `eval/out/` 裡既有的真模型紀錄、也沒有執行一套只需 6 秒的測試。
- 原因：宣稱與觀察的層級沒有對齊——看到檔案存在就下「功能通過」的結論，效力超出實際觀察到的層級；審查流程缺少強制的證據分級，讓最弱的證據得以支撐最強的結論。
- 解決方法：建立證據分級並強制標注——L0（檔案存在）／L1（讀過內容）／L2（執行觀察過行為）；L0 不得支撐「通過」；產物目錄必掃最新一份；交付前回掃全部「通過」項補齊證據等級。此紀律已寫入 design-review 協議。核心原則：**宣稱的效力止於實際觀察過的層級**。

**問題 2：npm audit 回報 undici 漏洞**
- 緣由：npm audit 回報前端依賴鏈中的 undici 存在已知漏洞。
- 原因：undici 是 devDependency 間接引入的套件，不會進入 build 後的部署產物，實際暴露面限於開發工具鏈；npm audit 預設不區分 dev／prod 依賴，需自行判讀影響範圍再決定處置強度。
- 解決方法：確認其屬 devDependency、不入部署產物後，以 npm audit fix 流程更新修補。原則：audit 結果先分流（部署產物 vs 開發工具鏈）再決定處置的急迫性。

## 學到的概念

- **證據分級 L0/L1/L2**：檔案存在（L0）≠ 內容正確（L1）≠ 行為正確（L2）；審查結論只能站在實際觀察過的層級上，跨級宣稱正是這次兩度改寫結論的來源。
- **審查與開發同樣需要可重複的紀律**：checklist、證據分級、產物目錄必掃，把「審查品質」從審查者當下的狀態中解耦——與 eval harness 把「模型品質」可量測化是同一件事。
