## 完成項目

### 1. 前端依賴整理與漏洞處置

處理 react-markdown 相關依賴問題，恢復前端 Markdown 預覽。另外處置安全掃描回報的漏洞：

- **問題**：npm 的安全掃描（npm audit）回報前端依賴鏈中的 undici（Node.js 的 HTTP 用戶端套件）存在已知漏洞。
- **原因**：undici 是開發期依賴（devDependency）間接引入的套件，不會進入建置後的部署產物，實際暴露面限於開發工具鏈；安全掃描預設不區分開發期／部署期依賴，需自行判讀影響範圍再決定處置強度。
- **解決**：確認其屬開發期依賴、不進部署產物後，以 npm audit fix 更新修補。原則：安全掃描的結果先分流（部署產物 vs 開發工具鏈）再決定處置的急迫性。

### 2. V0.2 需求逐條審查

依「雲端硬碟系統 需求與開發計畫書 V0.2」逐條審查 harness 模組，在兩個不同的對話中各獨立審一輪、再取聯集，共 9 項有效發現、0 項誤報。同步盤點五個驗證層的測試現況與驗收缺口。完整清單與驗證計畫見 [V02審查發現與測試現況](../02-筆記-harness/V02審查發現與測試現況.md)，簡述如下：

1. 隱私保護機制誤擋：使用者自帶金鑰、自己選了外部模型仍被拒絕外送，錯誤訊息還顯示成「憑證被拒」誤導排查。
2. 批次整理只處理第一頁：檔案超過 200 項時其餘沒被整理，卻回報成功。
3. 技能生成的觸發條件太窄：直接請求「幫我解壓縮」被回「做不到」。
4. 驗收證據缺口：沒有一份完整的全案例真模型評測報告。
5. 中文對話長度估算不準：以英文假設估算、高估 2–3 倍，長對話可能被無聲截斷。
6. 已存流程「重跑」不再次確認：含寫入的步驟直接執行，且此決策沒有寫入文件。
7. 保命快照失敗時不告知：快照建立失敗只記在日誌、照常執行寫入，使用者不知道該次操作沒有保護。
8. 設計文件寫了尚未實作的功能：「平行執行子任務」實際仍是一個接一個執行。
9. 內網位址寫死在程式碼：環境特定的設定值不該進版本庫。

- **問題**：第一輪審查完成後，對審查結果本身再查核一輪，發現多個「通過」判定的證據只到「檔案存在／搜尋得到關鍵字」的層級；補查四次、兩次改寫結論——漏看既有的真模型紀錄、也沒有執行一套只需 6 秒的測試。
- **原因**：宣稱與觀察的層級沒有對齊——看到檔案存在就下「功能通過」的結論，效力超出實際觀察到的層級；審查流程缺少強制的證據分級，讓最弱的證據得以支撐最強的結論。
- **解決**：建立證據分級並強制標注——L0（檔案存在）／L1（讀過內容）／L2（執行觀察過行為）；L0 不得支撐「通過」；產物目錄必掃最新一份；交付前回掃全部「通過」項補齊證據等級。核心原則：**宣稱的效力止於實際觀察過的層級**。

## 待釐清

- **修復順序**：9 項發現全部要修，其中三項動到 LLM 相關路徑，修改前後都需要量測對照、彼此會互相影響，先後順序待排——
    1. 隱私保護機制（privacy gate）誤擋（發現 1）：使用者自帶金鑰、自己選了外部模型，等於已同意把內容送出去，但隱私保護機制仍一律拒絕，且錯誤訊息顯示成「憑證被拒／離線」，誤導排查方向。
    2. 技能生成的觸發條件太窄（發現 3）：審查時實測，對話直接輸入「幫我解壓縮」，助理回「做不到」——觸發技能生成要求句中同時有「生成類的動詞＋目標」，直接的操作請求進不了生成流程，而內建技能又沒有解壓縮。
    3. 中文對話長度估算不準（發現 5）：把對話裁切進模型長度上限時，用「1 個 token 約等於 4 個字元」的英文假設來估中文，中文實際約 1–1.5 字元，長度被高估 2–3 倍，長對話可能被無聲截斷。

    其餘六項彼此獨立，依嚴重度排序即可。
- **需要團隊決策的一項**（發現 6，與上面工程問題性質不同）：已儲存的工作流程按「重跑」時不會再次跳出確認就直接執行。「點重跑就視同同意」這個解讀可以成立，但目前沒有寫進任何文件，與「絕不自動執行寫入操作」的嚴格解讀有張力——需與團隊確認並寫入文件。
