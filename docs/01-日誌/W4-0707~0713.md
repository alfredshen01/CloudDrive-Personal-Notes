## 遇到的問題

**問題 1：codegen 被拖進低溫後，產碼品質崩盤**
- 緣由：給 codegen 加 `response_format` 結構化保護後，真模型 A/B 顯示產碼品質 0/4——JSON 信封完好，裡面的程式碼是壞的。
- 原因：結構化路徑統一用低溫 0.2，這對「格式收斂」正確、對「寫程式」錯誤——codegen 需要較高溫度的全採樣空間。溫度被「是否結構化」決定，而不是被「任務類型」決定，是錯誤的耦合。
- 解決方法：per-call temperature 覆寫，codegen 用 0.8。原則沉澱：**temperature 依任務類型而非是否結構化**。

**問題 2：DEC-031/032 之後仍有殘餘跳針**
- 緣由：grammar 約束與 enum 上線後，量測仍出現跳針樣本，且延遲極不穩定（最大到 300s）。
- 原因：grammar 只約束最後答案，模型的「思考」段不受任何約束——殘餘跳針全部發生在思考段。
- 解決方法：DEC-033——planner 預設 `think:false`，以 A/B 實驗（60 樣本、只改思考開／關一個變數）支持：通過率 60% → 100%、跳針 3/10 → 0/10、平均延遲 92.4s → 8.6s（快約 10 倍）。結論：思考對這顆小模型的規劃品質無可量測貢獻，卻是跳針與延遲的全部來源。同時裸露出真正的天花板：雜訊清掉後，困難集 M3/M5 仍約 47%，失敗典型是「合法但語意錯——漏掉使用者要的寫入步驟」，歸類為模型能力限制（機制無法保證，只能靠 prompt 有限度改善）。

**問題 3：確認執行成功後，助理卻說「還在等確認」**
- 緣由：使用者實測——按下確認、workflow 執行成功後，追問「剛剛怎麼樣」，助理回「還在等你確認」。
- 原因：記憶 v1 只在 `/chat` 路徑寫對話歷史；手動確認走 `/confirm`，執行結果從沒寫回 session——對記憶而言，確認之後的世界不存在。
- 解決方法：`/confirm` 執行後把結果摘要寫回 session。這個 bug 是真實使用抓到的，不是測試——再次印證同源測試的盲點。

**問題 4：嚴謹回想案例 recall 0/5——一次完整的誤診與翻案**
- 緣由：多輪 eval 中，「列出檔案後下一輪回想檔名」的嚴謹回想案例 0/5；奇怪的是代理（改名）案例卻 5/5。
- 原因：第一步是**錯的**——憑閱讀 `summarise_results` 推論「輸出含 UUID → 超 200 字截斷 → 丟名」，並預設列檔已成功；依此假設改了摘要格式，重跑仍 0/5。改跑 verbose 診斷（印出實際回覆與記憶內容）才發現列檔根本失敗、尚無輸出可供截斷。真因是兩個疊加缺陷：主因——模型列根目錄傳 `parent_id="root"`（而非 null），`_optional_uuid` 把 "root" 當 UUID 解析而崩潰，list_items 失敗、記憶中無檔名；次因——即使列檔成功，摘要序列化原始 dict（UUID 佔滿 200 字預算）也會截斷丟名（即先前誤修的那項，被主因遮蔽）。
- 解決方法：兩者皆修（root 哨兵支援＋摘要萃取檔名），端到端真模型驗證：list_items 成功 → 記憶存入「3 items: …」→ 第二輪正確回想全部三個檔名 → recall 0/5 → 5/5、三案例綜合 15/15。方法論收穫：**憑讀碼向前推論、預設上游成功，是典型認知陷阱；印出 runtime 的真實診斷才是可信判準。**

**問題 5：「低溫較差」的溫度效應其實是假象**
- 緣由：修跳針前的量測看起來 0.6 溫度表現較好，形成「溫度影響正確率」的印象。
- 原因：think:false 後乾淨重跑（4 溫度 × 10 次）四溫度全 100%、0 跳針；對照 thinking 開（4 溫度 × 5 次）85–90% 且有跳針。先前的差異來自**跳針傾向**（低溫接近 greedy decoding，更容易進入 repetition loop），不是溫度對正確率的真實效應——混雜變數未清除前的結論不可信。
- 解決方法：把 `structured_temperature=0.2` 的定位改寫清楚：取「打破 greedy decoding repetition loop 的最小非零溫度」以求確定性，並非因高溫會失敗。

## 測試與驗證

- **E8 think:false A/B**：60 樣本、單一變數對照——通過率 60%→100%、跳針 3/10→0/10、延遲 92.4s→8.6s。planner 預設值的變更有數據支持。
- **codegen A/B**：低溫 0/4 崩盤 → per-call 0.8 恢復，信封與碼分開評。
- **多輪記憶 eval**：純對話指涉 5/5、工具結果指涉 5/5、嚴謹回想 0/5 →（雙修後）5/5；三案例 × 5 次 = 15/15，pass rate 100%（temp 0.2）。
- **工具結果承載方式 A/B**：`tool` 角色訊息 0/4（幻覺假檔名，chat template 不消化孤立 tool 訊息）vs assistant 文字 4/4 → 以 assistant 文字承載，順帶零 migration。
- **溫度再量測**：think:false 下 4 溫 × 10 全 100%、0 跳針；thinking 開 4 溫 × 5 為 85–90%。
- 品質關卡照常：每階段 unit／mypy／ruff 全綠；CI mypy 一次紅（測試函式缺型別註記）當日修掉。

## 學到的概念

- **模型限制 vs 工程缺口**：工程缺口做一次永久解決（值得做）；模型限制無機制能保證，只能以量測數據為據、靠 prompt 有限度改善。M3/M5 的 47% 屬後者——這把尺決定投入排序。
- **A/B 的紀律**：一次只改一個變數（think 開關）、樣本量要夠（60）、疑似退化起 baseline 對打。E8 的說服力全部來自這個紀律。
- **診斷先於修復**：recall 0/5 的第一輪修復是無效的，因為跳過了「印出 runtime 確認現場」這一步。假設要用觀察驗證，不能用更多假設疊加。
- **真實使用是不可替代的 oracle**：confirm 記憶缺口、斷線取消佔住 Ollama 併發槽，都是真人操作發現的——eval 案例覆蓋不到沒想到的路徑。
