## 完成項目

### 1. 防跳針修正第三輪（DEC-033：規劃器預設關閉思考段）

（DEC-033 是專案決策紀錄的編號，下同。「思考段」指模型正式作答前先產生的一段推理文字，thinking。）

- **問題**：前兩輪防跳針修正（DEC-031／032）上線後，量測仍出現跳針樣本，且延遲極不穩定（最長到 300 秒）。
- **原因**：格式約束（grammar）只管最後答案，思考段不受任何約束——殘餘跳針全部發生在思考段。
- **解決**：規劃器（planner）預設關閉思考段（`think:false`），以 A/B 實驗（60 個樣本、只改思考開／關這一個變數）支持：通過率 60% → 100%、跳針 10 次中 3 次 → 0 次、平均延遲 92.4 秒 → 8.6 秒（快約 10 倍）。結論：思考段對這顆小模型的規劃品質沒有可量測的貢獻，卻是跳針與延遲的全部來源。同時顯露出真正的天花板：雜訊清掉後，困難案例（M3 查詢＋寫入、M5 多步驟引用）通過率仍約 47%，典型失敗是「格式合法但語意錯——漏掉使用者要的寫入步驟」，歸類為模型能力限制（機制無法保證，只能靠 prompt 有限度改善）。

### 2. 自建 skill 程式碼生成（codegen）的格式約束保護

自建 skill 程式碼生成（專案內稱 codegen：使用者用自然語言描述需求，由 LLM 產生 skill 程式碼）是最後一個未受格式約束保護的模型呼叫點，本週補上。

- **問題**：加上格式約束後，真模型 A/B 對照顯示產出的程式碼 4 次全壞——JSON 外層格式完好，裡面的程式碼品質崩壞。
- **原因**：結構化路徑統一用低溫度 0.2，這對「收斂格式」正確、對「寫程式」錯誤——寫程式需要較高溫度的完整取樣空間。溫度被「是否結構化」決定，而不是被「任務類型」決定，是錯誤的綁定。
- **解決**：改成每次呼叫可各自指定溫度，codegen 用 0.8 後品質恢復。原則沉澱：**溫度依任務類型決定，而非依是否結構化**。

### 3. 對話記憶 v1＋多輪評測

助理可回讀多輪對話脈絡＋自動摘要工具執行結果；評測系統從單輪擴成多輪（先播種狀態、再串連同一場對話），讓「記不記得前文」變成可量測的指標。工具結果放進對話的形式以真模型 A/B 對照定案：放在助理的文字訊息裡（放在 API 規格的 `tool` 角色訊息中，gemma4 讀不懂）。同期撰寫記憶 v1 的提案文件。

- **問題**：使用者實測——按下確認、計畫執行成功後，追問「剛剛怎麼樣」，助理回「還在等你確認」。
- **原因**：記憶 v1 只在一般對話路徑寫入對話歷史；手動確認走另一條確認路徑，執行結果從沒寫回對話紀錄——對記憶而言，確認之後的世界不存在。
- **解決**：確認路徑執行後把結果摘要寫回對話紀錄。這個缺陷是真實使用抓到的，不是測試抓到的——再次印證與程式同源的測試有共同盲點。

### 4. 回想能力（recall）診斷閉環（0/5 → 5/5）

- **問題**：多輪評測中，「列出檔案後、下一輪要求回想檔名」的嚴格回想案例 5 次全錯；間接指涉（「把剛剛那個檔改名」）案例卻 5 次全對。
- **原因**：第一步的診斷是**錯的**——憑閱讀程式碼推論「工具輸出含冗長的檔案 id → 超過 200 字摘要上限被截斷 → 檔名被切掉」，並預設列檔操作已成功；依此假設改了摘要格式，重跑仍 5 次全錯。改開詳細診斷（印出模型實際回覆與記憶內容）才發現列檔根本沒成功、根本沒有輸出可供截斷。真因是兩個疊加的缺陷：主因——模型列根目錄時傳了 `"root"` 這個字串當資料夾 id（正確應是空值），id 解析直接失敗，列檔工具沒執行、記憶中自然沒有檔名；次因——即使列檔成功，摘要把原始資料原樣塞入（冗長 id 佔滿 200 字預算）也會把檔名擠掉（就是先前誤修的那項，被主因遮住）。
- **解決**：兩者都修——列檔／建立／搬移工具接受 `"root"` 這類根目錄代稱不再失敗、摘要改成萃取檔名。端到端真模型驗證：列檔成功 → 記憶存入檔名 → 第二輪正確回想全部三個檔名 → 嚴格回想 0/5 → 5/5、三種記憶案例綜合 15/15。方法論收穫：**憑讀程式碼向前推論、預設上游一定成功，是典型的認知陷阱；印出執行當下的真實狀態才是可信的判斷依據。**

### 5. 溫度再量測（釐清「溫度效應」假象）

- **問題**：修跳針之前的量測看起來溫度 0.6 表現較好，形成「溫度影響正確率」的印象。
- **原因**：關閉思考段後乾淨重跑（4 種溫度、各 10 次）四種溫度全部 100% 通過、0 跳針；對照組（思考段開啟，4 種溫度、各 5 次）通過率 85–90% 且有跳針。先前的差異來自跳針傾向——溫度越低越接近每步只選最高機率字的 greedy decoding，越容易掉入重複循環——不是溫度對正確率的真實效應。混雜變數沒清除之前的結論不可信。
- **解決**：把結構化請求的溫度設定（0.2）的定位改寫清楚：取「能打破重複循環的最小非零溫度」以求行為穩定，並非因為高溫會失敗。

## 測試與驗證

- **思考段開／關 A/B 對照**（eval 任務清單編號 E8）：60 個樣本、只改一個變數——通過率 60%→100%、跳針 10 次中 3 次→0 次、平均延遲 92.4 秒→8.6 秒。規劃器預設值的變更有數據支持。

- **溫度再量測**：關思考段後 4 種溫度各 10 次全部通過、0 跳針；開思考段 4 種溫度各 5 次為 85–90%。
- 品質關卡照常：每階段單元測試／型別檢查／lint 全綠；CI 的型別檢查一次失敗（測試函式缺型別註記）當日修掉。
