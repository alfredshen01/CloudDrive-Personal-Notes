### 3. 本機約束解碼補全（TDD，先紅後綠）

把 planner 的 json_schema 約束從「僅外部模型」補到本機 Ollama 路徑，並寫入三件防護測試（每次呼叫必帶 schema、首次合法計畫不觸發 repair loop、手寫 schema 與 Pydantic model 的 drift test）。

- **問題（一）**：`OllamaLLMClient` 收到完整 json_schema 信封後，payload 只寫入 `format: "json"`——schema 整個被丟棄，欄位／型別／required 全不保證，格式跑掉只能靠 repair loop 事後重問（慢、耗 token）。
- **原因**：兩層。其一，舊註解顯示當初把 `response_format` 當 on/off 開關用，schema 內容沒看就丟了。其二，形狀不匹配——上游流通的是 OpenAI 信封形 `{"type":"json_schema","json_schema":{...}}`，但 Ollama 的 `format` 參數只認裸 JSON Schema，原封傳遞本來就不可行，必須拆殼。
- **解決**：新增 `_to_ollama_format()` 從信封取出裸 schema（已是裸 schema 則原樣通過）；三個先紅後綠的測試鎖住行為（含「無 response_format 時不設 format 與 temperature」——一般聊天路徑不受影響）。防護欄明確不動：`validate_plan`＋repair loop 保留（schema 只保證形狀，擋不住語意錯誤）；手寫 schema 不換自動產生（Pydantic 欄位都有預設值，自動 schema 會沒有 required、約束更弱），改用 drift test 防漂移。

- **問題（二）**：寫「planner 每次呼叫都帶 schema」的防護測試，斷言失敗——`ModelRouter` 本機路徑呼叫時沒轉發 `response_format`。
- **原因**：W2 的修復範圍限定「只外部路徑」，本機 Ollama 從頭到尾收不到 schema——就算問題（一）修好拆殼也沒東西可拆。斷點在中間層，先前靠讀碼推測判斷錯誤，先寫會失敗的測試直接把它逼出來。
- **解決**：本機呼叫補上 `response_format` 轉發，一行修掉。這是 TDD 價值的直接示範：假設會誤導，紅測試不會。

### 4. 以執行結果回報＋有限度重規劃（DEC-029）

第 0 級：失敗時由程式從 StepResult 組合訊息，不再回 LLM 規劃時寫的 `plan.reply`。第 1 級：chat 快速路徑執行失敗才把真實觀察餵回 planner 重規劃一次。第 2 級 agentic loop 決定不做，理由完整記入 DEC-029。

- **問題**：workflow 執行失敗，使用者收到的仍是成功語氣的回覆；confirm／rerun 路徑更是固定回 "Workflow executed."，即使內部狀態已是 failed。
- **原因**：`plan.reply` 是 LLM 在「規劃時」寫的——當下一個步驟都還沒執行，那句話本質是**預測**；service 卻在執行完之後把它原樣回給使用者。架構把預測當報告，不是模型說謊。
- **解決**：`_compose_failure_message` 由程式從 StepResult（確定事實）組合訊息——哪步失敗＋原因、已完成哪幾步、其後未執行；三條執行路徑統一「全成功→原訊息，有失敗→一律由程式以執行結果組合回報」。再加第 1 級：失敗才 replan 一次，replan 輸入含逐步真實結果；兩道護欄——只允許全 read-only 計畫（權限不升級，否則事前確認閘被掏空）、絕不迴圈。replan 範圍只限 chat 快速路徑：confirm 的授權是「那份具體清單」、rerun 的授權是「固定配方」，偷換步驟都違反授權邊界。

### 5. executor 失敗隔離（DAG 第一階段，DEC-030）

一個步驟失敗不再全域中止；`StepResult` 加 `skipped` 三態、每步必有一筆結果；前端以 skip 圖示區別渲染；DEC-030 正式記錄「維持串行、並行延後」的理由。

- **問題**：5 檔壓縮第 1 個失敗，其餘 4 個無關步驟完全沒做、也沒有任何紀錄。
- **原因**：executor 遇第一個錯就全域中止；「有依賴才該被擋」和「無關步驟」沒有被區分。
- **解決**：合併顯式 `depends_on` 與引數 `from_step` 兩種依賴判斷；被擋→記 skipped（註明依賴哪步）、失敗→記錄後繼續。新不變量：每步必有一筆結果（ok／failed／skipped 三態）。並行明確延後（DEC-030）：所有技能共用同一 AsyncSession 不可並發是最硬前置，且隔離解決的是正確性、並行只省延遲，價值不對稱。

### 6. 防跳針循環一、二（DEC-031／DEC-032）

- **問題（一）**：某些請求（如「查剩餘容量」）讓本機模型陷入重複迴圈，卡到 300 秒逾時。
- **原因**：結構化請求把 temperature 釘 0，gemma4 在 greedy decoding 下掉入決定性的 repetition loop——同 prompt 100% 重現。
- **解決**：DEC-031 兩件事——`num_predict` 生成上限（就算跳針也是有界失敗，不再卡死）；結構化請求改用低而非零的溫度（微量隨機打破迴圈，格式仍由 grammar 保證）。原本卡死的整合測試 40/40 通過。

- **問題（二）**：E7 量測發現部分失敗案例是計畫含幻覺技能名，`validate_plan` 事後才攔截、觸發 repair 重問。
- **原因**：schema 只約束「skill 欄位是字串」，沒約束「是哪些字串」——亂編在文法上是允許的。
- **解決**：DEC-032——技能名做成 enum（依當下 registry 動態組進 schema），幻覺技能名在取樣層即不可生成。副發現：`validate_plan` 漏查 `depends_on` 非法相依，一併補上。

### 7. temp_sweep 量測工具＋E7 首次真模型量測

建立溫度掃描工具（代表案例 × N 次 × 多溫度，量 pass rate／跳針率／延遲），完成 E7 首次量測——首次量化確認破壞性規劃是模型重災區（0–40% 可靠度，跳針與幻覺技能並存），直接排定循環二、三的優先序。

- **問題**：首輪數據出現大量 401，且一次後端提早退出沒被偵測、整批數據作廢。
- **原因**：401 是 token 過期的假失敗（長時間掃描超過 access token 壽命）；後端子行程死掉時就緒檢查沒偵測行程存活。
- **解決**：每案例前重新登入；就緒檢查加子行程存活偵測。教訓：**量測工具本身也要驗**，否則資料驅動的迭代建立在髒數據上。
