## 遇到的問題

**問題 1：Ollama client 把 schema 降級成弱檔 JSON mode**
- 緣由：`OllamaLLMClient.chat()` 收到完整 json_schema 信封後，payload 只寫入 `format: "json"`，欄位／型別／required 全不保證，格式跑掉只能靠 planner 的 repair loop 事後重問（慢、燒 token）。
- 原因：兩層。其一，舊註解顯示當初把 `response_format` 當 on/off 開關用，schema 內容沒看就丟了。其二，形狀不匹配——上游流通的是 OpenAI 信封形 `{"type":"json_schema","json_schema":{...}}`，但 Ollama 的 `format` 參數只認裸 JSON Schema，原封傳遞本來就不可行，必須拆殼。
- 解決方法：新增 `_to_ollama_format()` 從信封取出裸 schema（已是裸 schema 則原樣通過）；三個先紅後綠的測試鎖住行為（含「無 response_format 時不設 format 與 temperature」——一般聊天路徑不受影響）。防護欄明確不動：`validate_plan`＋repair loop 保留（schema 只保證形狀，擋不住語意錯誤）；手寫 schema 不換 `model_json_schema()`（Pydantic 欄位都有預設值，自動產生的 schema 會沒有 required，約束更弱），改用 drift test 防漂移。

**問題 2：router 本機路徑漏傳 response_format（測試先紅抓出）**
- 緣由：寫「planner 每次呼叫都帶 schema」的防護測試，斷言失敗：`[None] == [_PLAN_RESPONSE_FORMAT]`。
- 原因：W2 的 json_schema 修復範圍限定「只外部路徑」，`ModelRouter` 呼叫本機 client 時沒轉發 `response_format`——本機 Ollama 從頭到尾收不到 schema，就算 ollama.py 修好拆殼也沒東西可拆。斷點在中間層，靠讀碼推測（「上游可能沒生成 schema」）判斷錯誤，先寫會失敗的測試直接把它逼出來。
- 解決方法：本機呼叫補上 `response_format=response_format`，一行修掉。這是 TDD 價值的直接示範：假設會騙人，紅測試不會。

**問題 3：執行失敗卻跟使用者說做完了**
- 緣由：workflow 執行失敗，使用者收到的仍是成功語氣的回覆；confirm／rerun 路徑更是固定回 "Workflow executed."，即使內部狀態已是 failed。
- 原因：`plan.reply` 是 LLM 在「規劃時」寫的——當下一個步驟都還沒執行，那句話本質是**預言**；service 卻在執行完之後把它原樣回給使用者。架構把預測當報告，不是模型說謊。
- 解決方法：`_compose_failure_message` 由程式從 StepResult（確定事實）組合訊息——哪步失敗＋原因、已完成哪幾步、其後未執行；三條執行路徑統一「全成功→原訊息，有失敗→一律由程式以執行結果組合回報」。再加第 1 級：失敗才 replan 一次，replan 輸入含逐步真實結果（LLM 這次看得到「search 回 0 筆」這種事實）；兩道護欄——只允許全 read-only 計畫（權限不升級，否則事前確認閘被掏空）、絕不迴圈。replan 範圍只限 chat 快速路徑：confirm 的授權是「那份具體清單」、rerun 的授權是「固定配方」，偷換步驟都違反授權邊界。

**問題 4：一個步驟失敗，其餘無關步驟全滅**
- 緣由：5 檔壓縮第 1 個失敗，其餘 4 個無關步驟完全沒做、也沒有任何紀錄。
- 原因：executor 遇第一個錯就全域 break；「有依賴才該被擋」和「無關步驟」沒有被區分。
- 解決方法：`_blocked_dependencies` 合併顯式 `depends_on` 與引數 `from_step` 兩種依賴判斷；被擋→記 skipped（註明依賴哪步）、失敗→記錄後 continue。新不變量：每步必有一筆結果（ok／failed／skipped 三態）。並行則明確延後（DEC-030）：所有技能共用同一 AsyncSession 不可並發是最硬前置，且隔離解決的是正確性、並行只省延遲——價值不對稱。

**問題 5：特定請求讓模型跳針卡死到 timeout**
- 緣由：某些請求（如「查剩餘容量」）讓本機模型陷入 repetition loop（重複迴圈），卡到 300s timeout。
- 原因：結構化請求把 temperature 釘 0，gemma4 在 greedy decoding 下掉入決定性的 repetition loop——同 prompt 100% 重現。
- 解決方法：DEC-031 兩件事：`num_predict` 生成上限（就算跳針也是有界失敗，不再卡死）；結構化請求改用低而非零的溫度（微量隨機打破 repetition loop，格式仍由 grammar 保證）。原本卡死的整合測試 40/40 通過。

**問題 6：模型規劃出不存在的技能名**
- 緣由：E7 量測發現部分失敗案例是計畫含幻覺技能名，`validate_plan` 事後才攔截、觸發 repair 重問。
- 原因：schema 只約束「skill 欄位是字串」，沒約束「是哪些字串」——亂編在文法上是允許的。
- 解決方法：DEC-032——技能名做成 enum（依當下 registry 動態組進 schema），幻覺技能名在取樣層即不可生成。這是「機制優先於模型自覺」的教科書例：不是求模型別亂編，是讓亂編文法上不可能。

**問題 7：量測工具本身有 bug，數據先被污染**
- 緣由：temp_sweep 首輪數據出現大量 401，且一次後端提早退出沒被偵測、整批數據作廢。
- 原因：401 是 token 過期的假失敗（長時間掃描超過 access token 壽命）；後端子行程死掉時 `_wait_ready` 沒檢查行程存活。
- 解決方法：每案例前重新登入；`_wait_ready` 加子行程存活檢查。教訓：**量測工具本身也要驗**，否則資料驅動的迭代建立在髒數據上。

## 學到的概念

- **機制 vs 模型自覺**：能用文法（grammar／enum）或程式（validate_plan、執行結果回報）保證的，不靠模型自律。本週三件事（約束解碼、執行結果回報、enum）全是同一原則的實例。
- **授權邊界**：replan 等於在使用者視線外執行新計畫，只在「授權給規則（read-only 免確認）」處合法；授權給「具體清單」（confirm）或「固定配方」（rerun）的路徑不可偷換步驟。
- **機率性系統不看單次**：temp_sweep 的設計前提——代表案例 × N 次 × 多溫度，看 pass rate 分佈而非單次成敗。
- **誠實記錄「暫不做」**：DEC-029（不做 agentic loop）、DEC-030（不做並行）都是把「不做什麼、為什麼」寫成正式決策，比假裝完整更有價值。

## 本週小結

本週是 harness 可靠性工程的主體週：把結構化輸出補到本機路徑、把回報從「預言」改成「事實」、把失敗從「全滅」改成「隔離」、把跳針與幻覺技能用機制根治或有界化，並建好 temp_sweep 讓後續迭代有快速量測。四條線共同構成「量化→定位根因→修正→再驗證」方法論的第一次完整運轉。
