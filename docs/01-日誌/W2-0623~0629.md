# W2（2026-06-23 – 06-29）

## 本週主軸

- 把 CloudDrive 開發環境完整建起來，熟悉前端架構。
- 為 AI 助理推進四項功能：模型選擇、自建 skill 進對話、結構化輸出修復、多組具名連線。
- 建立執行稽核（execution audit）方法：查核 AI 宣稱的操作是否真的執行。

## 完成項目

### 1. 開發環境建置與前端架構導讀

把 CloudDrive 在本機完整跑起來（Postgres 容器、後端、前端 dev server），並通讀前端架構。

### 2. 助理功能：模型選擇

對話框加模型下拉，選定即只用該模型、不自動 fallback；`OllamaLLMClient` 加 `connect_timeout=5s`，連不到的本機秒失敗（原本每次對話卡 300 秒逾時，是做這個功能的直接動機）；chat 端點錯誤分類（接不到／憑證被拒或額度／其他）。

### 3. 助理功能：自建 skill 可在對話中使用（chat-skills）

使用者自建的 skill（如 `compress_to_zip`）可被 planner 選用——一律 write 級（必經確認閘）、沙盒執行、以勾選檔案帶入目標（多選逐檔展開）、每個 skill 預設關閉需手動啟用。

- **問題**：對話框輸入「用我的 skill 壓縮檔案」，助理回「技能集中沒有這個功能」。
- **原因**：planner 的技能清單只載入 13 個內建 skill；自建 skill 存在 DB 裡（一段程式文字），原本只能經右鍵選單觸發沙盒執行，從未被餵進 planner 的 prompt。
- **解決**：把已安裝且允許進對話的 skill 以 write 級＋沙盒橋接 handler 註冊進 registry；名稱與內建衝突則跳過並提示改名。「對哪個檔案操作」不靠 LLM 猜，由前端勾選檔案帶入、逐檔展開步驟。

### 4. 助理功能：外部模型結構化輸出修復（json_schema）

外部模型走 planner 時改用 `response_format: json_schema`。

- **問題**：選 Gemini 後助理只會閒聊——回「好的，我會將選取的檔案壓縮成 zip」之類的話，但確認閘不出現、什麼都沒執行。
- **原因**：planner 要求 LLM 回固定 JSON `{"reply", "steps":[...]}`。本機 Gemma 靠 prompt 就照做，Gemini 不照——三種模式實測：純文字回自然語言（解析失敗變閒聊）；`json_object` 回 JSON 但欄位名錯；只有 `json_schema` 完全正確。另有前置缺陷：planner 組 prompt 時沒告知「使用者已勾選 N 個檔案」，Gemini 一直反問要對哪個檔案操作。
- **解決**：chat 介面加 `response_format` 參數，由 planner → ModelRouter → 外部 client 串進 payload；schema 不加 strict（strict 與開放的 arguments 物件衝突，OpenAI 相容端點會拒收）；planner 加系統訊息告知選取數量。範圍限定「只動 planner、只動外部路徑」——這個限定在 W3 被證明留下缺口（本機路徑漏傳，見 W3）。

### 5. 助理功能：多組具名模型連線

以新資料表取代單一憑證：每筆連線有 label／kind／base_url／model／key，助理下拉列出本機＋每筆連線，額度用完可直接切換另一筆。

### 6. 外部模型實測（Gemini、Ollama cloud）

實測兩家服務的可用組態與限流行為，把可用組態與模型目錄記進筆記。

- **問題**：測試中 Gemini 開始回 429 RESOURCE_EXHAUSTED；Ollama cloud 連測多次後所有模型回 401 Unauthorized。
- **原因**：Gemini 免費方案每天每模型只有 20 次請求（各模型額度獨立），當日測試加診斷就用光；Ollama cloud 免費 key 有限流，401 是限流不是設定錯。另有組態陷阱：Ollama cloud 要用 `openai_compatible` kind＋`https://ollama.com/v1`（原生 `/api/chat` 路徑會 404 且不保證 json_schema）。
- **解決**：換模型取得獨立額度（`gemini-2.5-flash-lite`）、等額度重置。長期結論：穩定測試需要付費、或多備來源輪替。

### 7. 執行稽核：AI 回報與實際執行的查核方法

助理宣稱「刪好了」但檔案原封不動——本項建立查核方法，確認宣稱的刪除從未發生。

- **問題**：請助理「把選取的檔案丟到垃圾桶」，回覆「做好了」，實際什麼都沒做。
- **原因**：planner 回的 `reply` 是給人看的話、`steps` 才是要執行的動作，兩者是同一次 LLM 輸出的不同欄位——LLM 可能 reply 寫「刪好了」但 steps 是空的，此時 service 直接回覆那句話、不執行也不留紀錄。查核確認 trash 工具本身存在，是 planner 沒把「刪除」對應到它——LLM 規劃失敗，而非缺工具。
- **解決**：本週的產出是查核方法而非修復——用 psql 對三張表交叉查核：`assistant_workflows`（7 筆全是壓縮、無任何刪除計畫）、`assistant_workflow_runs`（只有真執行才有）、`activity_logs`（trash 事件 0 筆）。確立法則：**LLM 的自述不能當證據，真做了必留 DB 稽核紀錄；兩者都沒有＝沒做**。（「回報必須以實際執行結果為依據」在 W3 以「回覆由程式從執行結果組合」的機制（honest reporting）根治。）

### 8. 檔案預覽／下載缺陷診斷（repo 既有 bug，本週僅診斷）

- **問題**：點檔案「預覽」顯示 Failed to load preview，點「下載」回 401 Not authenticated。
- **原因**：兩個獨立缺陷。其一，前端打的預覽資訊路徑後端沒有註冊，對不上回 404；單元測試沒抓到是因為前端 MSW mock 了這條後端不存在的路徑，假後端把真實契約不一致蓋住了。其二，下載與預覽直連 URL，瀏覽器不帶 Authorization header，而 access token 只存記憶體（不放 cookie），後端回 401。
- **解決**：本週先不修——用 git 比對確認是 repo 本身的 bug（與 origin 一字不差），並用帶 token 的 curl 得到 200 證明端點正常。修法方向記入待辦。（隊友於 6/29 以 blob 載入方式實作修復，7/13 隨 PR #10 合併。）

## 待釐清

- **連線網址未設防護（SSRF 風險）**：「多組具名連線」的 base_url 可以填任意網址，後端會直接對它發請求——惡意使用者可填內網位址，讓伺服器代為存取內部服務（SSRF，伺服器端請求偽造）。防護方向：網址白名單、或僅允許對外的 https 位址。
- **選檔與計畫畫面顯示內部 id 而非檔名**：系統內部以 UUID 識別檔案（檔名可重複、也可能被改名，id 唯一且穩定，所以操作一律用 id），但顯示層直接把 id 露給使用者是第一版未完成的部分——選取狀態只存了 id 沒存名稱，要顯示檔名需擴充狀態或查快取，待改善。
- **計畫確認畫面的前端流程尚未實際走過**：寫入／破壞性計畫會標記為待確認、由使用者在前端按下確認才執行；本週只從後端確認了這個機制存在，還沒實際操作過前端的確認畫面。
