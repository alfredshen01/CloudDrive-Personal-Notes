## 遇到的問題

**問題 1：選 Gemini 後助理只會閒聊，不跳確認、不執行**
- 緣由：對話框選外部模型（Gemini）後，助理回「好的，我會將選取的檔案壓縮成 zip」之類的話，但確認閘不出現、什麼都沒執行。
- 原因：planner 要求 LLM 回固定 JSON `{"reply", "steps":[...]}`。本機 Gemma 靠 prompt 就照做，Gemini 不照——三次實測確認：純文字模式回自然語言（解析失敗變閒聊）；`json_object` 模式回 JSON 但欄位名錯（`step`/`args`）；只有 `json_schema` 嚴格結構化輸出完全正確。另有一個前置缺陷：planner 組 prompt 時沒告知「使用者已勾選 N 個檔案」，Gemini 一直反問要對哪個檔案操作。
- 解決方法：chat 介面加 `response_format` 參數，由 planner → ModelRouter → `ExternalLLMClient` 串進 payload；schema 不加 strict（strict 與開放的 arguments 物件衝突，OpenAI 相容端點會拒收）；`planner.plan(selected_count=...)` 加系統訊息告知選取數量。範圍嚴格限定：只動 planner、只動外部路徑（這個限定後來在 W3 被證明留下缺口——本機路徑漏傳，見 W3 問題 2）。

**問題 2：對話中叫不動自建 skill**
- 緣由：對話框輸入「用我的 skill 壓縮檔案」，助理回「技能集中沒有這個功能」。
- 原因：planner 的技能清單只載入 13 個內建 skill；使用者自建／安裝的 skill 存在 DB 的 `assistant_skills.code`（一段文字），原本只能經右鍵選單觸發沙盒執行，從未被餵進 planner 的 prompt。
- 解決方法：`_register_installed_skills` 把 installed 且 `chat_enabled` 的 skill 以 write 級＋`requires_selection`＋沙盒橋接 handler 註冊進 registry；名稱與內建衝突則跳過並提示改名。「對哪個檔案操作」不靠 LLM 猜，由前端勾選檔案帶入、`expand_selection_steps` 逐檔展開步驟。

**問題 3：檔案預覽 404、下載 401（repo bug，本週僅診斷、未修）**
- 緣由：測助理時點檔案「預覽」顯示 Failed to load preview，點「下載」回 `{"detail":"Not authenticated"}`。
- 原因：兩個獨立缺陷。Bug 1——前端打 `GET /preview/{id}/info`，後端只註冊 `/preview/{id}` 與 `/preview/{id}/content`，路徑對不上回 404；單元測試沒抓到是因為前端 MSW mock 了 `/preview/{id}/info`，假後端把真實契約不一致蓋住了。Bug 2——下載用 `<a href>`、預覽用 `<iframe src>` 直連 URL，瀏覽器不帶 Authorization header，而專案設計 access token 只存在記憶體（不放 cookie），後端 `HTTPBearer` 回 401。
- 解決方法：本週先不修——已用 git 比對確認是 repo 本身的 bug（與 origin 一字不差、非本機環境造成），並用帶 token 的 curl 打 `/download/{id}` 得到 200 證明端點本身正常。修法方向記錄待辦：Bug 1 前端改打 `/preview/{id}` 對齊後端；Bug 2 改 Axios 抓 blob（會帶 token）再產 object URL。（後續由隊友在 PR #10 以 blob 載入方式實作。）

**問題 4：助理宣稱「刪好了」，實際上什麼都沒做**
- 緣由：對話框請助理「把選取的檔案丟到垃圾桶」，助理回「做好了」，但檔案原封不動。
- 原因：planner 回的 `reply` 是給人看的話、`steps` 才是要執行的動作，兩者是同一次 LLM 輸出的不同欄位——LLM 完全可能 reply 寫「刪好了」但 steps 給空的，此時 service 看到空 steps 就直接回覆那句話、不執行也不留紀錄。追查確認 trash 工具本身存在（`trash_item`，destructive 級），是 planner 沒把「刪除」對應到它——LLM 規劃失敗，而非缺工具。
- 解決方法：本週的產出是**查證方法**而非修復——用 psql 對三張表互相印證：`assistant_workflows`（7 筆全是壓縮、無任何刪除計畫）、`assistant_workflow_runs`（只有真執行才有）、`activity_logs`（trash 事件 0 筆、有史以來只有 create × 7）。確立法則：LLM 的自述不能當證據，真做了必留 DB 稽核紀錄；兩者都沒有＝沒做。（「回報必須以實際執行結果為依據」在 W3 以「回覆由程式從執行結果組合」的機制（honest reporting）根治。）

**問題 5：外部模型免費額度與限流，卡住測試**
- 緣由：測試中 Gemini 開始回 429 RESOURCE_EXHAUSTED；Ollama cloud 連測多次後所有模型回 401 Unauthorized。
- 原因：Gemini free tier 每天每模型只有 20 次請求（各模型額度獨立），當日測試加診斷探測就用光；Ollama cloud 免費 key 有限流，401 是限流不是設定錯。另踩到組態雷：Ollama cloud 要用 `openai_compatible` kind＋`https://ollama.com/v1`（原生 `/api/chat` 路徑會 404 且不保證 json_schema）。
- 解決方法：換模型取得獨立額度（`gemini-2.5-flash-lite`）、等額度重置；把可用組態與模型目錄記進筆記。長期結論：穩定測試要嘛付費、要嘛多備來源輪替——這也是後續把量測重心放在本機模型的原因之一。

## 學到的概念

- **registry 與沙盒的分工**：registry 是 `name→skill` 字典，LLM 只「點名」、registry 查表呼叫 handler；自建 skill 的 handler 是橋接到沙盒（把 DB 裡的程式文字寫成暫存 .py、`python -I` 隔離子行程執行），不可信程式碼只拿原檔唯讀複本、只能寫自己的 output_dir、禁網路禁子行程；「上傳結果」由沙盒外的可信層做——功能成立靠分層，不靠放寬沙盒。
- **「OpenAI 相容」是協定不是廠商**：OpenAI／Gemini／Ollama／Groq 多半提供 `/chat/completions`，差別只在 base_url＋model＋key——多連線功能的設計基礎。
- **DB 稽核紀錄 vs runtime log**：執行紀錄放 DB 而非 log 檔是為了交易一致（刪檔和寫紀錄同 transaction，要嘛都成要嘛一起 rollback），所以能拿來對帳。
- **planner 只規劃、不執行**：`reply` 是預測性文字，`steps` 才進 executor；理解這個分層是後續所有「以執行結果回報」工作的前提。

## 本週小結

本週把助理從「只能用內建功能、模型寫死」推進到「可選模型、可用自建 skill、外部模型格式可靠」，同時透過 DB 對帳第一次直面「LLM 自述不可信」的問題。json_schema 與對帳兩件事共同指向同一個主題：不靠模型自覺，靠機制與證據。
