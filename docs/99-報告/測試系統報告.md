# CloudDrive AI 助理：測試與驗證系統設計

本文件說明 CloudDrive 內嵌 AI 助理之測試與驗證系統的設計動機、量測方法、遇到的問題、
改良方式與結果。全文以「為什麼需要此測試 → 問題 → 改良 → 結果」為敘事主軸。
測試環境、案例 ID 與原始數據索引見文末附錄 A–C。

---

## 1. 為什麼需要專屬的驗證系統（動機）

一個以大型語言模型（LLM）為核心的系統，無法僅以傳統單元測試保證正確性。單元測試中的
LLM 為 mock（預先寫好的假回應），測試通過只證明「程式管線接得正確」，不證明「真實模型
的行為正確」。


因此本專案建立一套「真實模型」的驗證系統，自外部呼叫 `/assistant/chat`，對真實的
gemma4:26b 量測其行為，而非僅驗證程式邏輯。此為全文所有測試的共同動機。

## 2. 測試系統的組成與方法論

### 2.1 三層測試

- **底層——確定性單元／整合測試**（mock LLM）：CI 快速執行，防止程式管線退化。
- **中層——LLM 考官（judge）**：由另一模型對回應給分並列出優缺點（質性評估）。
- **頂層——真實模型 eval sweep**：以真實 gemma4 實跑，量測通過率、延遲、跳針率。

【圖表 1：三層測試金字塔】

**說明**：本報告所有量化結果（通過率）皆由底層之確定性驗證器判定，非由中層考官打分；
考官為輔助之質性能力（見 §5）。

### 2.2 eval Harness 元件與執行模式

eval harness（位於 `backend/eval/`）由下列元件組成：`schema`（案例定義）→ `runner`
（驅動執行）→ `verifier`（斷言）→ `scoring`（評分），並旁掛 `judge`（考官）與
`baseline`（回歸比較）。支援三種執行模式：**API**（呼叫 HTTP 端點）、**browser**
（以 Playwright 驅動真實使用者介面）、**exec**（實際執行生成的技能並驗證產出）。

【圖表 2：eval Harness 元件與三種執行模式】

### 2.3 確定性驗證器如何判定正誤

驗證器不以模型打分，而是將模型回傳的結構化 JSON 進行 `==` 與 `in` 的比對。以「改名需
確認、計畫須含 rename_item」之案例為例，模型回傳 `{plan:{status, steps:[...]}}`，驗證器
執行下列判定：

```python
status == "pending_approval"           # 計畫是否標記為需確認
"rename_item" in json.dumps(steps)     # 計畫是否含指定技能
"RenamedByAgent" in json.dumps(steps)  # 計畫是否含指定新名
```

三項皆為真則通過。相同的回應必得相同的判定結果，不似考官打分具變異性。「規劃為唯讀」
此一失敗型態亦由此機械判定：期望值為 `pending_approval`，實際值為 `auto_executed`
（意即計畫中不含任何寫入步驟，因唯讀計畫才會自動執行），單一比對即判為失敗，並記錄
實際的技能清單供分類。

### 2.4 量測方法論

本系統遵循「量化 → 定位根因 → 修正 → 再驗證」之迭代流程：

- 對機率性系統不以單次結果為準，以 `--runs N` 量測通過率。
- 疑似退化時，另起 baseline 版本後端於獨立埠對照同一請求，取得基準後再定位。
- 診斷時先將完整日誌落檔，再行過濾。

## 3. 測試線一：規劃可靠性（防跳針）

### 3.1 動機

部分請求（如查詢剩餘容量）會使助理停滯——整合測試整體凍結、無法完成。需要一套可重複
量測「跳針率、延遲、通過率」的工具，方能以數據驅動修正。

（**跳針**：模型重複生成同一段內容且無法停止之現象，以下沿用此稱。）

### 3.2 遇到的問題

- **跳針導致逾時**：模型陷入重複生成，持續產出至讀取逾時（300 秒）而回傳 503。此為「停滯」
  之機制。
- **溫度似有影響**：首次掃描（thinking 開啟）顯示 0.6 溫度較佳、破壞性案例於各溫度皆差，
  易誤判為溫度問題。
- **破壞性規劃為模型弱點**：可靠度 0–40%，跳針與幻覺技能（規劃出不存在的技能名）並存。
- **量測工具本身之缺陷**：單一溫度區塊被跳針樣本拖延逾 30 分鐘，導致存取權杖過期、後續
  案例全數回傳假性 401（數據污染）。

### 3.3 改良方式（三層縱深防護與量測工具）

建立 `temp_sweep` 真實模型量測工具，並依數據做出三項決策：

| 決策 | 改良 | 解決之環節 |
|---|---|---|
| DEC-031 | `num_predict` 生成上限 + 結構化請求採低而非零之溫度 | 跳針達上限即停止，以快速失敗取代 300 秒停滯（有界失敗） |
| DEC-032 | 計畫 schema 之技能名以 enum 枚舉（依 registry 動態組成） | 幻覺技能於取樣層即無法生成 |
| DEC-033 | planner 預設 `think:false` | 移除跳針之唯一棲息地（思考段），使跳針無從發生 |

- **三層縱深邏輯**：使其不拖垮系統（有界失敗）→ 使其有界失敗 → 使其不發生。
- **名詞定義**：**thinking** 指 gemma 此類模型在輸出正式答案前先生成的一段內部推理；
  **grammar 約束（結構化解碼）** 指將 JSON schema 編譯為文法、於取樣時遮蔽不合法之 token，
  使輸出必然符合格式。
- **跳針為何位於思考段**：grammar 僅約束最終之結構化答案，無法約束思考段（自由文字）；
  思考段於低溫貪婪解碼下最易陷入重複迴圈。`think:false` 直接移除此段。
- **量測工具之修正**：每案例執行前重新登入（修正權杖過期造成之假性 401），並偵測後端
  是否提早退出。

### 3.4 結果

**實驗一：think:false A/B 對照**（代表案例 × 5 次，僅變更思考開／關單一變數）

| 組別 | 通過率 | 跳針 | 平均延遲 |
|---|---|---|---|
| thinking 開啟（對照） | 60% | 3/10 | 92.4 秒 |
| think:false（實驗） | 100% | 0/10 | 8.6 秒（約快 10 倍） |

**實驗二：溫度掃描**（4 案例，thinking 開啟為每溫度 20 樣本、think:false 為每溫度 40 樣本）

| 溫度 | thinking 開啟 通過率 | think:false 通過率 |
|---|---|---|
| 0.2 | 90% | 100% |

thinking 開啟另有跳針 1–4/20、平均延遲 44–66 秒、最大延遲達 300 秒；think:false 則

【圖表 4：溫度掃描 thinking 開啟 vs think:false 之通過率對比折線】

- **溫度之判讀**：修正跳針前所見之「溫度效應」，實為跳針傾向（低溫貪婪解碼更易陷入迴圈）；
  `think:false` 移除思考段後，各溫度通過率一致為 100%，可知溫度並非正確率之直接因素。故將
  `structured_temperature` 固定於 0.2（能打破 greedy decoding repetition loop 之最小非零溫度），以求輸出之確定性，
  而非因高溫會失敗。
- **模型能力上限**：雜訊清除後，困難案例集 M3/M5 之通過率仍僅約 47%。（eval 案例分為 M1–M5，
  難度遞增；M3/M5 為最難之兩級：先以多個查詢工具蒐集脈絡，再執行一個寫入動作。）失敗之典型
  型態為「合法但語意錯誤」——遺漏使用者要求之寫入步驟（規劃為唯讀）。此屬模型能力之限制，
  無法以機制解決。

## 4. 測試線二：對話記憶之可量測化

### 4.1 動機

新增「對話記憶」功能後，缺乏可重現之指涉測試，無法證明「模型確實使用了對話歷史」。需將
「多輪指涉」轉為可量測之數字，補上此驗證缺口。

### 4.2 遇到的問題

- **多輪失憶**（功能缺陷）：對話已寫入資料庫，但規劃器僅收到當前訊息，歷史從未回讀。
- **工具結果之承載形式**：置入歷史之結果，模型能否解讀？
- **確認執行未寫回**（使用者實測發現）：確認執行成功後詢問「剛才執行如何」，回覆「仍在
  等待確認」。
- **回想全數失敗（recall 0/5）**：嚴謹回想案例全數失敗——診斷後為兩個疊加之缺陷（列根目錄
  崩潰 + 摘要截斷，詳見 §4.4）。

### 4.3 改良方式

- **記憶接線**：規劃器載入最近 N 則歷史；工具結果經真實模型 A/B 決定以 assistant 文字承載
  （`tool` 角色之訊息 gemma4 無法解讀，0/4 對比 assistant 文字 4/4）。
- **confirm 修復**：`/confirm` 執行後將結果摘要寫回 session。
- **多輪 eval**：擴充 harness 以支援狀態播種與 session 串接，新增三個指涉案例與 `reply_contains`
  （驗證回覆文字內容）。
- **recall 雙修復**（診斷後，見 §4.4）：其一，`_optional_uuid` 接受根目錄哨兵（`root` 等）並
  視為 None，使列檔不再崩潰；其二，摘要對集合型輸出萃取檔名，不再序列化原始 dict。

### 4.4 結果與一項「eval 發現、真實使用未觸及」之缺陷

| 案例 ID | 驗證內容 | 結果 |
|---|---|---|
| `multiturn-create-second` | 自對話文字解析「第二個 = 2023」 | 5/5 |
| `multiturn-rename-first` | 自結果摘要取得 item_id 執行改名（代理） | 5/5 |
| `multiturn-recall-listed-names` | 僅自對話報出全部三個名稱（嚴謹） | 修復前 0/5 → 修復後 5/5 |

【圖表 7：多輪記憶 eval 三案例通過率（回想案例標示修復前後）】

**代理測試與嚴謹測試之別（為何需要兩個案例）**：改名案例僅驗證「計畫含 rename_item 與新名」，
此間接證明模型使用了歷史中的某個 id（改名必須提供 id，其唯一來源為上一輪之摘要），但未嚴格
驗證「使用了正確且完整的內容」，故稱代理（proxy）測試。回想案例則直接要求模型報出全部三個
名稱，屬嚴謹測試。關鍵在於：改名（代理）通過反而遮蔽了列檔失敗（其不需真正列出名稱亦能生成
改名計畫），是嚴謹回想案例方將缺陷暴露——此說明寬鬆之代理測試會遺漏缺陷，須以嚴謹案例把關。

**此缺陷之情境**：全新對話 → 「列出我的資料夾」（列根目錄）→ 下一輪「剛才那些的名稱為何」。

**真實使用未觸及之原因**（此為「為何需要 eval」之最佳實例）：
- 使用者查看檔案時直接使用檔案總管介面，不會特意要求助理「列出根目錄」；對助理之請求多為
  針對已勾選檔案之動作（使用真實 item_id，不經此路徑）。
- 「列出後再詢問名稱」此類回想操作於日常幾乎不會發生。
- 因此缺陷潛伏於「經助理列根目錄並回想」此一真實使用甚少經過之路徑，直至 eval 刻意執行方
  暴露。此即 eval 之價值：系統化地執行真實使用不會自然觸及、卻確實存在缺陷的角落。

**診斷過程（方法論實例）**：初步將 0/5 判為「摘要截斷」，此為憑閱讀程式碼之推論、未經診斷
（且預設列檔已成功）。依此假設修改摘要格式後重跑仍為 0/5；執行 verbose 診斷（印出實際回覆）
後方發現兩個疊加之缺陷：其一（主因），模型列根目錄時傳入 `parent_id="root"`（而非 null），
`_optional_uuid` 試圖將 "root" 解析為 UUID 而崩潰，導致 list_items 失敗、記憶中無檔名；其二
（次因），即使列檔成功，摘要序列化原始 dict（含 36 字元之 UUID）亦會超出 200 字元上限而被
截斷。

**修復與驗證**：兩者皆修正後，真實模型重跑之 recall 由 0/5 提升至 5/5，三案例綜合通過率
100%（15/15）、0 跳針。此結果印證「真實診斷方為可信之判準；憑閱讀程式碼向前推論、預設上游
成功，是典型之認知陷阱」。

## 5. 限制

- **單一模型／單一 GPU**：結論綁定於 gemma4:26b，更換模型須重新量測。
- **樣本數**：部分 sweep 之樣本數為 5–10，通過率反映趨勢而非精確值。
- **合成案例與真實路徑覆蓋**：eval 案例為刻意構造，擅長暴露真實使用甚少經過之角落缺陷
  （如 recall 案例），但案例本身不等同真實使用之分佈。
- **考官變異與判定分工**：本報告所有通過率皆由確定性驗證器判定，非由考官打分；考官具主觀性，
  故其定位為輔助之質性評估。

## 6. 未來方向

- **擴充案例**：增加指涉型案例與執行次數以提升解析度。
- **考官升級**：將 judge 更換為能力更強之 provider（E6 已具備此設計）以降低變異。
- **量測前沿問題**：以多輪 eval 為工具，迭代 planner 對寫入意圖之提示工程，嘗試推進 M3/M5
  之 47% 上限。
