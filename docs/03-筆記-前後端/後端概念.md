## 資料庫

### Alembic migration

- 每次改 schema 寫一個版本檔，部署時 `alembic upgrade head` 自動套用還沒跑過的版本。
- `alembic stamp <revision>`：把指定版本標記為已完成、不實際執行 SQL——用於「資料表已用其他方式建立過」的第一次接入（手動建的表沒有 `alembic_version` 記錄，Alembic 會誤以為資料庫全新而重跑所有 migration）。
- 鐵則：**已套用過的 migration 永遠不回頭改或刪**，要變更就往前加新的，否則 revision 鏈會壞掉。
- 只做加法（`ADD COLUMN`、`CREATE TABLE`）是安全的；刪欄位、改名是破壞性操作要謹慎。
- schema 只能有一個真實來源：曾因 `initdb/*.sql`（死檔）與 alembic 並存造成混亂，確認 alembic 是唯一來源後刪除前者。

### 外鍵與 CASCADE

- 被引用的列不能直接刪（`violates foreign key constraint`），要先刪子資料再刪父列。
- 建表時設 `ondelete=CASCADE` 可讓父列刪除時自動清掉子列（例：刪使用者連帶清其檔案記錄）。

### pgvector 與語意搜尋

- `pgvector` 是 Postgres 的 extension，新增 `vector` 欄位型別，支援相似度搜尋。
- CloudDrive 的用法：上傳檔案 → embedding 模型（Ollama `nomic-embed-text`，768 維）把內容轉成向量 → 存 `file_embeddings` 表 → 自然語言提問也轉向量 → 找語意最接近的檔案（不是關鍵字比對）。
- 注意：`postgres:16-alpine` image **沒有** pgvector，要用 `pgvector/pgvector:pg16`。

### DB 稽核紀錄 vs 執行期日誌

- 執行期日誌（runtime log）是程式跑的時候印在終端機的訊息——重啟就消失、也可能漏印，不能當證據。
- 執行紀錄放進資料庫則有**交易一致性**保證：刪檔和寫紀錄綁在同一個資料庫交易（transaction）裡，兩者只會同時成功或同時失敗，不會出現「檔刪了卻沒紀錄」或「有紀錄卻沒刪」——所以 DB 紀錄可以拿來做執行稽核；另外可查詢、可關聯、不怕重啟。

### 執行稽核：三表交叉查核 LLM 是否真的做事

助理宣稱「做好了」不能當證據。用 psql 直查三張表交叉查核（連線帳密見 `backend/.env`，不寫入筆記）：

| 表 | 意義 |
|---|---|
| `assistant_workflows` | 計畫本身與狀態（pending_approval／executed／cancelled） |
| `assistant_workflow_runs` | **只有真的執行才有這筆**，`step_results` 記每步 ok／error |
| `activity_logs` | drive 層級 CRUD 事件，`item_id` 連到 `drive_items` |

法則：真做了 → 有 `workflow_runs`＋對應的 `activity_logs` 事件；兩者都沒有＝沒做。LLM 產生的檔案（如壓出的 zip）經 upload 流程寫回硬碟後也是一等公民的 `drive_items`、各有 UUID。

## 測試

### 單元測試 vs 整合測試

- 單元測試：不需要真資料庫，用 Mock 假裝 DB，測單一模組邏輯，數百個測試幾秒跑完。
- 整合測試：需要真的 Postgres（`tests/integration/`），測多模組串起來的完整流程；用獨立的 `clouddrive_test` 資料庫，跑之前設 `DATABASE_URL` 指向它。
- 教訓（前端同理）：mock 會把「真實契約不一致」蓋住——preview 路徑 404 的 bug 沒被測出來，就是因為 MSW mock 了一條後端根本沒有的路徑。同源測試與程式共享盲點，真實使用才是 oracle。

## Docker 與後端的關係

- Image 是打包好的環境快照、Container 是它的執行實例；`docker run` 是「怎麼跑起來」，與 image 本身是兩件事。
- 自己的程式（FastAPI）直接在主機跑、外部依賴服務（Postgres）用容器跑，是常見的開發組合；共通點只是「都是長時間執行的服務」。
- port 映射 `0.0.0.0:5434->5432/tcp`：左邊是主機 port（外面的程式連這個）、右邊是容器內 Postgres 真正監聽的 port；IPv4 與 IPv6 各綁一次所以列兩行。
- Docker 權限：操作都經 `/var/run/docker.sock`，只有 root 或 `docker` 群組可存取；`sudo usermod -aG docker $USER` 後要登出再登入才生效。

## HTTP 檔案傳輸

- HTTP 回應標頭只能用 latin-1 編碼——中文檔名放進 `Content-Disposition` 會拋 `UnicodeEncodeError` 變 500。正解是 RFC 5987 寫法：ASCII 安全的 `filename` 當備援＋UTF-8 百分比編碼的 `filename*`。

## 認證

### JWT 與 bcrypt 的分工

| | bcrypt hash | JWT token |
|---|---|---|
| 用途 | 驗證你是誰（密碼） | 驗證你已登入（通行證） |
| 存放位置 | 資料庫 users 表 | 客戶端 |
| 何時產生 | 註冊時 | 登入成功後 |
| 永久性 | 永久（除非改密碼） | 暫時（到期失效） |
| 用到的 key | 無（演算法本身） | `SECRET_KEY`（簽名） |

- bcrypt 是單向的：驗證是「把輸入重新 hash 再比對」，不是解開 hash；明文密碼只存在註冊那一瞬間。
- JWT 用 `SECRET_KEY` 簽名防偽造；換掉 `SECRET_KEY` 只讓舊 token 失效，密碼 hash 不受影響。
- CloudDrive 的進階設計：access token 只存前端記憶體、refresh token 走 HttpOnly cookie 且 DB 只存 hash。
