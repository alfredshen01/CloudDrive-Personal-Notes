# 開發筆記精選

> 蒸餾自 cloud_drive/local-docs/note.md（2026-06-25 ~ 07-05 逐日開發筆記）的可重用結論，並補入 W4 的 temperature 結論。每條為固定欄位：原則／緣由／依據／一句話記住。逐日完整脈絡見 [W2](../01-日誌/W2-0623~0629.md)～[W4](../01-日誌/W4-0707~0713.md) 日誌。

## 1. 結構化輸出（json_schema）

### 1.1 跨模型的結構化輸出必須用 schema 級約束

- **原則**：要 LLM 回固定 JSON，prompt 懇求不算數，必須 `json_schema` 級約束。
- **緣由**：Gemini 走 planner 時，純文字模式回自然語言（解析失敗變閒聊）、`json_object` 模式欄位名錯（`step`/`args`）。
- **依據**：三種模式實測對照，只有 `json_schema` 完全正確。
- **一句話記住**：格式要靠文法保證，不靠模型聽話。

### 1.2 傳遞鏈每一層都要轉發

- **原則**：`response_format` 從 planner → ModelRouter → 各 LLM client，每層都要顯式轉發；修一條路徑時列舉所有路徑。
- **緣由**：W2 只修外部路徑，router 本機路徑漏傳，本機 Ollama 從頭到尾收不到 schema。
- **依據**：防護測試（斷言每次呼叫必帶 schema）先紅抓出漏傳，一行修掉。
- **一句話記住**：假設會誤導，紅測試不會。

### 1.3 信封形狀不匹配要拆殼

- **原則**：上游流通 OpenAI 信封形 `{"type":"json_schema","json_schema":{...}}`，Ollama 的 `format` 參數只認裸 JSON Schema——原封傳遞不可行，必須取出裸 schema。
- **緣由**：舊程式把 `response_format` 當 on/off 開關，schema 內容沒看就丟棄，降級成弱檔 JSON mode。
- **依據**：`_to_ollama_format()` 拆殼後三個先紅後綠的測試鎖住行為。
- **一句話記住**：跨協定傳遞前，先確認兩端認的形狀相同。

### 1.4 schema 的兩個工程取捨

- **原則**：①schema 不加 strict（strict 與開放的 arguments 物件衝突，OpenAI 相容端點會拒收）；②手寫 schema 不換 `model_json_schema()`（Pydantic 欄位都有預設值，自動產生的 schema 沒有 required、約束更弱），改用 drift test 防漂移。
- **一句話記住**：更嚴或更自動不一定更好，取捨要有理由並用測試鎖住。

### 1.5 schema 只保證形狀，語意驗證是第二道防線

- **原則**：schema 擋不住幻覺技能名、缺參數——`validate_plan`＋repair loop 不可拆；技能名進一步做成 enum（依當下 registry 動態組），讓亂編在文法上不可生成。
- **一句話記住**：形狀用 grammar 保證，語意用程式驗證。

## 2. 回報與授權

### 2.1 失敗回報由程式組合，不沿用模型的話（honest reporting，DEC-029）

- **原則**：失敗時的回覆由程式從 StepResult（確定事實）組合，不回模型規劃時寫的 `plan.reply`。
- **緣由**：`plan.reply` 是規劃當下寫的，本質是預言；執行完把它原樣回給使用者＝把預測當報告——「沒做完卻說做完」的架構根因，不是模型說謊。
- **依據**：三條執行路徑統一「全成功→原訊息，有失敗→程式組合的事實回報」，TDD 先紅後綠。
- **一句話記住**：預言不能當報告。

### 2.2 replan 的合法性由授權來源決定

- **原則**：replan＝在使用者視線外執行新計畫，只在「授權給規則」處合法。chat 快速路徑授權給「read-only 免確認」規則 → 可 replan（護欄:權限不升級、只重試一次）;confirm 授權給「那份具體清單」、rerun 授權給「固定配方」→ 偷換步驟即掏空確認閘，只做事實回報。
- **一句話記住**：授權給規則可以重試，授權給清單不能換內容。

### 2.3 模型的自述不能當證據

- **原則**：真做了必留 DB 稽核紀錄；自述與紀錄不符時以紀錄為準（三表對帳見 [後端概念](後端概念.md)）。
- **一句話記住**：查紀錄，不聽轉述。

### 2.4 「不做」也要正式記錄

- **原則**：決定不做 agentic loop（DEC-029）連同理由入決策紀錄：權限模型依賴完整計畫先行、小模型長程 loop 易漂移、成功案例不該付 N 倍呼叫成本、留翻案條件。
- **一句話記住**：把「不做什麼、為什麼」寫成決策，比假裝完整更有價值。

## 3. 失敗隔離

- **原則**：executor 不因一步失敗全域中止。不變量：每步必有一筆結果（ok／failed／skipped 三態）、skipped ⇒ 有不可用上游依賴、ok ⇒ 依賴全 ok；依賴判斷合併顯式 `depends_on` 與引數 `from_step`；「引用不存在的 index」是 failed 不是 skipped。
- **緣由**：原本 5 檔壓縮第 1 個失敗，其餘 4 個無關步驟全沒做、也無紀錄。
- **依據**：hypothesis property 測試（200 例）鎖住不變量。
- **並行延後的理由（DEC-030）**：所有技能共用同一 AsyncSession 不可並發是最硬前置；隔離解正確性、並行只省延遲，價值不對稱。
- **一句話記住**：無關的步驟不陪葬,被擋的步驟留紀錄。

## 4. temperature 依任務類型

- **原則**：temperature 由任務類型決定，不由「是否結構化」決定——格式收斂用低溫、寫程式用高溫（codegen per-call 0.8）。
- **緣由**：結構化路徑統一低溫 0.2，codegen 被拖進去後產碼品質崩盤（JSON 信封完好、碼壞）。
- **依據**：真模型 A/B 0/4 → 0.8 恢復；`structured_temperature=0.2` 的定位是「打破 greedy decoding repetition loop 的最小非零溫度」，非因高溫會失敗（think:false 後 4 溫度全 100%）。
- **一句話記住**：低溫收斂格式，高溫寫程式。

## 5. registry 與沙盒的分層

- **原則**：registry 是 `name→skill` 字典，LLM 只「點名」、registry 查表呼叫 handler；自建 skill 是 DB 裡的程式文字，執行時由沙盒寫成暫存 .py、`python -I` 隔離跑；不可信程式碼只拿原檔唯讀複本、只能寫自己的 output_dir、禁網路禁子行程；「上傳結果」由沙盒外的可信層做。
- **接進對話的安全預設**：自建 skill 一律 write 級（必跳確認）、預設關閉需手動啟用、「對哪個檔操作」由前端勾選注入而非讓 LLM 猜。
- **一句話記住**：功能成立靠分層，不靠放寬沙盒。

## 6. 外部模型實測與限流

### 6.1 「OpenAI 相容」是協定不是廠商

- **原則**：OpenAI／Gemini／Ollama／Groq 多半提供 `/chat/completions`，差別只在 base_url＋model＋key——「多組具名連線」設計的基礎。
- **一句話記住**：換模型來源是換三個參數，不是換架構。

### 6.2 Gemini free tier

- **事實**：每天每模型 20 次請求、各模型額度獨立（換模型有全新額度）；偶發 503 high demand 屬暫時性、重試即可；429 RESOURCE_EXHAUSTED＝當日額度用光，key 本身沒壞。

### 6.3 Ollama cloud

- **事實**：要用 `openai_compatible` kind＋`https://ollama.com/v1`（原生 `/api/chat` 配 `/v1` base_url 會 404，且不保證 json_schema）；模型名要用 `/v1/models` 目錄內名稱；免費 key 有限流——連測多次後全模型回 401 是限流不是設定錯，等重置即恢復。

### 6.4 結論

- **原則**：外部免費額度撐不起穩定測試——要嘛付費、要嘛多備來源輪替;這是把量測重心放在本機模型的原因之一。
- **一句話記住**：穩定量測的前提是穩定的額度。
