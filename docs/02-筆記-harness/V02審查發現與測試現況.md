# V0.2 需求審查發現與測試現況

> 2026-07-16 依「雲端硬碟系統 需求與開發計畫書 V0.2」逐條審查 Harness Agent（我負責的 In-App AI 助理模組）的結果整理。
> 審查方法：design-review checklist（逐條判定＋證據分級 L0 存在／L1 內容／L2 行為）；由兩個獨立 session 各審一輪後取聯集，共 9 項有效發現、0 項誤報。

## 一、符合度總結

FR-18～FR-21、NFR-07、5.6 節 harness 組件、5.7 節助理操作前自動 snapshot、第八章 eval harness——全部有對應實作，主線設計正確。以下 9 項為需要處理的缺口，依嚴重度排列。

## 二、9 項發現

| # | 問題 | 原因說明 | 嚴重度 |
|---|---|---|---|
| 1 | 隱私閘擋掉「明確選擇的外部模型」 | 使用者自帶金鑰、明選外部模型，等同已同意外送；但預設 `privacy_default=sensitive` 時 `ModelRouter._call_external` 仍一律拒絕，且錯誤訊息顯示為「憑證被拒／離線」，與真因不符，誤導排查方向 | 中 |
| 2 | `organize_by_type` 只整理第一頁 200 項 | 只呼叫 `list_items(page=1, page_size=200)`，根目錄超過 200 項時其餘檔案不被整理，回報卻顯示成功 | 中 |
| 3 | 直接說「幫我解壓縮」得到「做不到」 | skill 生成的意圖偵測要求句中同時含「生成類動詞＋目標詞」；直呼操作句進不了 authoring 流程，planner 又無內建解壓 skill | 中低 |
| 4 | 驗收缺「全案例真模型」報告 | 驗收標準要求「eval harness 通過」，現有真模型紀錄為分散的部分案例，無一份完整全案例報告 | 中低 |
| 5 | context 裁切對中文高估 2~3 倍 | `ContextManager` 以「1 token ≈ 4 字元」估算，中文實際約 1~1.5 字元／token，長對話可能超出模型視窗被靜默截斷 | 低 |
| 6 | 已存 workflow「重跑」不再次確認 | 含寫入的步驟直接執行；「點重跑即同意」可以成立，但此決策未寫入文件，與 FR-19「絕不自動執行」的嚴格解讀有張力 | 低 |
| 7 | 保命 snapshot 失敗時靜默放行 | snapshot 建立失敗只記 log、照常執行寫入，使用者不知道該次操作沒有保命 snapshot | 低 |
| 8 | 「有界平行子任務」寫在設計但未實作 | executor 實際為循序執行；tasks 文件已列為待辦，但 V0.2 5.6 節讀起來像已具備 | 低 |
| 9 | 內網位址寫死在程式碼預設值 | `config.py` 預設 `llm_base_url` 為實際內網位址，環境特定值不應進版本庫 | 低 |

## 四、驗證計畫（依序執行）

1. 起環境：`docker compose up postgres`＋後端 dev server，確認 Ollama 可達；註冊測試帳號取得 token。
2. 補全案例真模型 eval（填第 4 項「驗收證據缺口」）：
   ```bash
   cd backend
   uv run python -m eval.run --llm real --base-url http://localhost:8000 \
     --token <TOKEN> --runs 3 --tag acceptance --save-baseline eval/out/baseline_acceptance.json
   ```
   exec 類案例另以 `--mode exec` 執行；含 rubric 的案例加 `--judge`。多 runs 取 pass rate，不以單次論斷；報告存 `eval/out/` 作為驗收證據並留 baseline 供回歸比對。
3. 執行 E2E：`npm run test:e2e`（至少 assistant spec）。
4. 真人走一遍情境三完整生命週期，逐步核對文件承諾。
5. 修復紀律：每項先寫可重現的測試再修；第 1、3、5 項動到 LLM 路徑，修畢以步驟 2 的 baseline 做 A/B 對照。

## 五、審查方法的教訓（過程紀錄）

第一輪審查有多個「通過」判定的證據只到「檔案存在／grep 命中」層級，後設稽核補查四次、兩次改寫結論（漏看 `eval/out/` 既有真模型紀錄；未執行只需 6 秒的測試套件）。修正方式已寫入 design-review 協議：證據分級 L0（存在）／L1（讀過內容）／L2（執行觀察）強制標注，L0 不得支撐「通過」；產物目錄必掃最新一份；交付前回掃全部「通過」補齊等級。核心原則：宣稱的效力止於實際觀察過的層級。
