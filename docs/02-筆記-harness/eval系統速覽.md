# Eval 系統速覽

> 一頁版。完整的動機、方法、數據與附錄見 [測試系統報告](../99-報告/測試系統報告.md)（正式版以 docx 為準）；實驗的時間脈絡見 [harness筆記](harness筆記.md) 第二部。

## 目標

- **是什麼**：以「真實模型」的行為量測取代「只信單元測試」，保證 AI 助理的可靠度。
- **為什麼**：單元測試中 LLM 是 mock，全綠只證明「管線接得對」，不證明「真模型做得對」——實例：codegen 加結構化輸出後 615 個單元測試全綠，真模型量測產碼品質 0/4。
- **一句話記住**：可重複量測 → 資料驅動修正 → 逼出真實使用甚少經過的缺陷角落。

## 系統組成

- 位置 `backend/eval/`，**離線**、從外部呼叫 `/assistant/chat`，把助理當黑箱，不在正式請求路徑上。
- 元件：`schema`（案例定義）→ `runner`（驅動）→ `verifier`（斷言）→ `scoring`（評分），旁掛 `judge`（LLM 考官，質性輔助）與 `baseline`（回歸比較）。
- 三層測試：底層 確定性單元／整合測試（mock，CI 用）；中層 LLM 考官打分（輔助）；頂層 真模型 eval sweep（所有量化結果來源）。
- 三種執行模式:**API**（HTTP）、**browser**（Playwright 驅動真 UI）、**exec**（實跑生成的技能驗產出）。

## 怎麼自動判對錯（機械判定，非模型打分）

驗證器把模型回傳的結構化 JSON 做 `==`／`in` 比對，同樣回應必得同樣判定：

```python
status == "pending_approval"           # 是否標記為需確認
"rename_item" in json.dumps(steps)     # 計畫是否含指定技能
```

失敗型態「規劃成唯讀」即由此量到：期望 `pending_approval`、實得 `auto_executed`（計畫無任何寫入步驟）。

## 邊界與限制

- **邊界**：eval 擁有「案例／驅動／判定／評分」；驅動但不擁有「助理、真模型、DB、考官」;介面是外部 HTTP——如同外部考官：出卷、監考、改卷,不擁有考生。
- **不在範圍**:不訓練模型、不跨模型（綁定 gemma4:26b）、不代表真實使用分佈、非負載/安全測試。
- **限制**：部分 sweep 樣本 5–10（通過率為趨勢非精確值）；考官主觀故僅輔助。
