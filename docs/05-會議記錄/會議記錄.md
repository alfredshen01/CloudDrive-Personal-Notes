本次為進度回饋（細節於後續會議再談）。整體評語：目前看起來不錯、進度一般。討論與建議：

1. **容器隔離 vs uv 虛擬環境**：若已用容器做隔離，是否還需要 uv 虛擬環境？可從「用了有什麼優缺點」「uv 出現前大家怎麼做」切入。參考關鍵詞：PEP 668、conda / pip / uv 比較、CI/CD、虛擬化、容器化、podman、kubernetes（不需短期內全部理解）。此題沒有標準答案，看當下取捨，可再討論。
2. **資料庫部署形式**：目前 PostgreSQL 裝在本機（VM）上，可討論「資料庫是否應以容器形式部署」。
3. **合作階段主題發想**：可開始構思合作階段想嘗試的主題（例如實作、復現或部署某篇有興趣領域論文的想法）。

**補充（回應「API 與資料庫之間的溝通算不算 API」）**：

- 從廣義的電腦科學角度算，但日常溝通通常不這樣稱呼。
- 在 Web 開發語境中，工程師口頭說「API」通常特指 Web API（RESTful、GraphQL、gRPC），也就是透過 HTTP／HTTPS 暴露給前端、手機 App 或第三方呼叫的端點（endpoint）。
- 資料庫的溝通通常說「後端透過 driver 建立資料庫連線（DB connection）」，或「透過 ORM 下 SQL 查詢」，而不說「後端去打資料庫的 API」。
- 結論：第一階段練習提到的「API」，更精確的說法是「Web API」；建議延伸了解 TCP/IP model（網路概念）與 ORM（API server 實作）。

## 2026-06-08 · 威廷、晉緯、德偉學長

**Part I　專題（小組）**

- 現況：架構鬆散、缺乏模組化 workflow。
- 先釐清使用者面向（persona）與使用情境（scenario），從需求反推架構。
- 先定義「要解決什麼問題」→ 再實作實用的 API → 最後評估實用性與便利性。
- 非必要處避免用 LLM：省 token，且把流程固化更可靠。
- 限制使用者的 AI 使用權限：過度開放會有安全疑慮（prompt injection、資料外洩、不可控行為）。
- 成本：需節制 API token 的用量。

**Part II　LLM Wiki 問題（個人）**

*一、摘要式知識庫的使用情境*

- 優點（相對 RAG）：知識能複利與增長（每次 ingest 都能與既有知識融合、更新）、省 token（不必每次對多個 chunk 做 RAG）。
- 缺點：純摘要式的 raw data 對處理文件的使用者夠用，但研究／coding 場景可能遺失關鍵資訊。
- 依用途分化形式：當筆記用 → 需把文本視覺化；當 coding／研究的資料庫用 → 只靠摘要恐不足。

*二、多資料庫架構*

- 問題：目前 LLM wiki 的用法偏 coding 使用；為避免整個資料庫的資料分散，需自建不同專業的資料庫，手動放入對應庫再做 ingest。
- 設計方向：統一大入口負責 ingest；ingest 時自動分類 → 歸到對應資料庫。
- 僅在重要技術介面處做跨庫關聯（如前後端協作的交集點），其餘各自獨立。
- 待想：模組化 workflow 怎麼設計、使用者怎麼用。

*三、其他問題*

- Wiki 與 RAG 結合：Wiki 當高層索引／導覽，底下指向原文 chunk（RAG），同時解決可讀性與完整性。
- RAG 背後的 chunking 策略怎麼做。
- 自動分類的可靠度。
- 將流程模組化，並非每次搜尋都要做精確的 RAG 檢索。
